Making a connection

R users can establish connections to a SQL server database using the DBI and ODBC packages. The first step to establishing a connection is to build a connection string. A connection string contains a set of key value pairs that tells R where and with what credentials to make a connection. Different driver types require diffrent connection string formats. Here are two common connection string configurations:

```{r commonCon, eval=FALSE}

myDriver <- ‘SQL Server’ # Localhost can be referred to with . myServer <- ‘.\snapman’ myDatabase <- ‘Cab_Demo’

trusted_connection <- DBI::dbConnect(odbc::odbc(),Driver= myDriver,Server = myServer,Database = myDatabase,Trusted_Connection=‘yes’)

myUserid <- ‘Cab_App’ myPassword <- Sys.getenv(‘Cab_App_Password’)

connection <- DBI::dbConnect(odbc::odbc(),Driver= myDriver,Server = myServer,Database = myDatabase,Uid = myUserid,Pwd = myPassword)

```

I don’t need to pass a user name and password to the first connection string trusted_connection because the argument “Trusted_Connection” enables the use of my current windows account. Linux clients can use the second string and with a user name and password. It’s also good practice to avoid storing clear text passwords in the client code or in blog posts, hence I stored an enviromental variable and accessed it with the Sys.getenv function.

Common Connection Problems

I’ve encountered a fair share of error when attempting the intial connection. They usually stem from one or more of the following:

  • My connection string isn’t properly formatted. ConnectionStrings.com is a good place to check for that my connection string is well formed. I botch the server name a lot. Also, I struggle with spelling in general which means a lot of mispelled values.
  • I don’t have permissions to the SQL Server. If I suspect permissions are an issue than I usually try to connect through SSMS first. When I see errors like the “The server principal… is not able to access…” then I suspect it’s permissions.
    If you’re connecting to a server controlled by IT then the system administrator should be able to troubleshoot permission issues. If you’re working at a large enterprise then it may be a challenge to track down the system admin. I’ve always had luck starting with a helpdesk support ticket. If you own the server then check the SQL server logs through SSMS. SQL Server logs failed logins by default.
  • I don’t have the right driver selected.
  • I don’t have a driver installed. One can check installed drivers from the ODBC Data Source Adminstrator > Drivers tab.
  • I need to specify the port number. If the default port 1433 has been changed and is fixed then I need to specify the port number after the server name e.g. myServer = ‘.\snapman,2050’
  • A firewall is blocking the connection. One of my previous blog posts shows how to configure a firewall SQL Server access.
